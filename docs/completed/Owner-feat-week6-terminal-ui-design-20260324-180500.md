## Problem Statement

lil_bro requires LibreHardwareMonitor (LHM) for thermal monitoring — one of its
highest-value features for the target user (casual gamers who just upgraded hardware).
Currently, if LHM isn't pre-installed, the tool degrades silently: "Thermal monitoring
unavailable. Install from: https://...". The target user (non-technical casual gamer)
will never install a separate dependency. Thermals must work out of the box.

## Demand Evidence

- Diffuse but consistent: repeated pattern of misconfigured gaming PCs observed across
  Discord/Reddit/friend groups — users running hot without knowing it
- Status quo: users Google individual thermal problems and follow one-off YouTube/Reddit
  guides. No single tool catches all misconfigurations in one pass.
- Thermals are the check most likely to prevent hardware damage or throttling — high
  stakes for someone who just spent $1,500+ on a gaming PC.

## Status Quo

Users either:
1. Don't monitor thermals at all (most common)
2. Install LHM or HWMonitor manually after Googling "why is my PC hot"
3. Watch one YouTube video about fan curves and call it done

lil_bro's value: one scan catches thermal issues alongside refresh rates, power plans,
XMP, Game Mode, and everything else — but only if thermals actually work.

## Target User & Narrowest Wedge

**Target:** Casual gamer who just bought a prebuilt or upgraded a component. Their
mindset: "am I getting what I paid for?" They'll download one tool and run it once.
They will NOT install a separate .NET application as a prerequisite.

**Narrowest wedge:** Ship a custom minimal sensor server binary alongside `lil_bro.exe`
that provides the exact same HTTP `/data.json` API that LHM provides. Zero additional
user steps — thermals work on first run.

## Constraints

1. **Privacy-first:** No external calls beyond what's already permitted (driver version checks)
2. **Admin required:** Sensor access requires admin privileges — already handled (lil_bro runs as admin via UAC manifest)
3. **Windows only:** LibreHardwareMonitorLib only works on Windows — matches lil_bro's platform
4. **Portable .exe:** Must work in PyInstaller onefile mode — companion binary extracted alongside or to %APPDATA%
5. **MPL-2.0 license:** LibreHardwareMonitorLib is MPL-2.0. Using it as a NuGet dependency in a separate binary is permitted. Must include license attribution.
6. **.NET runtime:** Self-contained publish bundles the .NET runtime — no user-side .NET install required

## Premises

1. Thermal monitoring is high-value for the target user and shouldn't be optional
2. The current "install LHM yourself" UX is a dealbreaker for casual users
3. MPL-2.0 allows using LibreHardwareMonitorLib as a dependency with attribution
4. A custom minimal binary gives the most control and smallest footprint
5. Matching the existing HTTP API format means zero changes to the Python thermal pipeline

## Approaches Considered

### Approach A: Bundle LHM Portable (Companion Binary)
Include the full LHM portable release (~5MB extracted) in the build. The existing
sidecar code already searches `tools/LibreHardwareMonitor/`. Smallest change, but
ships the full GUI app (tray icon, settings, graphing — all unused).

**Rejected because:** Ships unnecessary GUI baggage. Tray icon appears briefly. User
sees a full "LibreHardwareMonitor" process in Task Manager — confusing for a tool
that's supposed to be a seamless part of lil_bro.

### Approach B: pythonnet + LibreHardwareMonitorLib.dll (In-Process)
Use the HardwareMonitor PyPI package to call sensors directly from Python via
pythonnet/.NET interop. No subprocess, no HTTP.

**Rejected because:** pythonnet + PyInstaller has documented compatibility issues.
Adds ~50MB dependency. .NET CLR crash takes down lil_bro. Requires rewriting the
entire thermal pipeline (lhm_sidecar.py, thermal_monitor.py).

### Approach C: Custom Minimal Sensor Server (.NET Console App)
Write a C# console app (~100-150 lines) that references LibreHardwareMonitorLib
via NuGet, opens temperature sensors, and serves JSON on localhost:8085 at `/data.json`
replicating LHM's exact tree structure. Build as a self-contained single-file .NET publish.

## Recommended Approach

**Approach C: Custom Minimal Sensor Server**

### Architecture

```
tools/
  lhm-server/
    LhmServer.csproj          — .NET 8.0 console app, references LibreHardwareMonitorLib NuGet
    Program.cs                — Minimal HTTP server serving /data.json on port 8085
    LICENSE-LHM.txt           — MPL-2.0 license text (attribution)
    build.ps1                 — Build script: dotnet publish → single-file self-contained
    reference-output.json     — Captured LHM /data.json snapshot for format verification
    dist/
      lhm-server.exe          — Output: ~10-20MB self-contained Windows x64 binary
```

### C# Sensor Server Design

The server does exactly three things:
1. **Open sensors:** Use `Computer` class from LibreHardwareMonitorLib with `IsCpuEnabled = true`, `IsGpuEnabled = true`, `IsMotherboardEnabled = true`
2. **Serve HTTP:** Minimal `HttpListener` on `http://localhost:8085/` responding to `/data.json`
3. **Format JSON:** Replicate the exact tree structure LHM's built-in web server uses. The Python parser (`_parse_temps_from_lhm()`) walks: root → `Children[]` → hardware nodes → `Children[]` → subhardware/sensor-type nodes → a node with `Text == "Temperatures"` → `Children[]` where each child has `Text` (sensor name), `Value` (formatted string like `"72.3 °C"`), and `RawValue` (numeric float like `72.3`).

**Required JSON output format (example):**
```json
{
  "Text": "Sensor",
  "Children": [{
    "Text": "Intel Core i7-14700K",
    "Children": [{
      "Text": "Temperatures",
      "Children": [
        {"Text": "CPU Package", "Value": "72.3 °C", "RawValue": 72.3},
        {"Text": "CPU Core #1", "Value": "70.1 °C", "RawValue": 70.1}
      ]
    }]
  }, {
    "Text": "NVIDIA GeForce RTX 4070",
    "Children": [{
      "Text": "Temperatures",
      "Children": [
        {"Text": "GPU Core", "Value": "65.0 °C", "RawValue": 65.0}
      ]
    }]
  }]
}
```

**Critical:** `RawValue` must be a JSON number (float), not a string. The Python parser calls `float(raw)` on it directly (thermal_monitor.py line 31).

### Prerequisite: capture reference JSON

Before writing the C# server, capture a real LHM `/data.json` snapshot from a running
LHM instance and save it as `tools/lhm-server/reference-output.json`. The C# server's
output must pass a structural diff test against this reference — same nesting depth,
same field names, same types. This is a blocking prerequisite, not an open question.

### Key implementation details

- **Sensor update loop:** Call `computer.Hardware[i].Update()` every 1 second in a background thread. HTTP handler reads latest values.
- **Graceful shutdown:** Listen for `Console.CancelKeyPress` and parent process termination. Clean up sensor handles.
- **No GUI:** Pure console app. No tray icon, no window (launched with `CREATE_NO_WINDOW`).
- **Self-contained publish:** `dotnet publish -r win-x64 -c Release --self-contained -p:PublishSingleFile=true -p:IncludeNativeLibrariesForSelfExtract=true -p:EnableCompressionInSingleFile=true` — bundles .NET runtime so users don't need .NET installed.
- **Kernel driver:** LibreHardwareMonitorLib uses a ring-0 kernel driver (`WinRing0x64.sys`) for low-level sensor access, extracted to a temp directory at runtime. See "Feasibility Risks" section below.

### Python-side changes

**`lhm_sidecar.py`** — Changes required:

1. **Search paths:** Add two new paths at the top of `_LHM_SEARCH_PATHS`:

```python
import sys
from ...utils.paths import get_appdata_dir

_LHM_SEARCH_PATHS = [
    # 1. PyInstaller frozen bundle (sys._MEIPASS / "tools")
    os.path.join(getattr(sys, '_MEIPASS', os.path.abspath('.')), "tools", "lhm-server.exe"),
    # 2. %APPDATA%\lil_bro\tools (future: first-run extraction)
    os.path.join(str(get_appdata_dir()), "tools", "lhm-server.exe"),
    # 3. Dev/source tree
    os.path.join(
        os.path.abspath(os.path.join(os.path.dirname(__file__), "..", "..", "..")),
        "tools", "lhm-server", "dist", "lhm-server.exe",
    ),
    # ... existing LHM full-install paths follow ...
]
```

2. **Binary detection:** Refactor `find_lhm_executable()` to return a tuple indicating which type of binary was found:

```python
def find_lhm_executable() -> tuple[str | None, bool]:
    """Search paths for a thermal server binary.
    Returns (path, is_custom_server). is_custom_server=True for lhm-server.exe."""
    for path in _LHM_SEARCH_PATHS:
        if os.path.isfile(path):
            is_custom = os.path.basename(path) == "lhm-server.exe"
            return path, is_custom
    return None, False
```

3. **Launch command:** In `start()`, skip the `--http-port` arg for the custom server (it has the port hardcoded):

```python
exe, is_custom = find_lhm_executable()  # was: exe = ... or find_lhm_executable()
if is_custom:
    cmd = [exe]  # lhm-server.exe — port hardcoded
else:
    cmd = [exe, "--http-port", str(LHM_PORT)]  # full LHM
self._process = subprocess.Popen(cmd, ...)
```

**Note:** The port check in `start()` already runs before `find_lhm_executable()` — no change needed there. The existing fallback chain order is preserved.

**`build.py`** — Add a `build_lhm_server()` step before `run_pyinstaller()`:

```python
def build_lhm_server():
    """Build the custom thermal sensor server (requires .NET 8 SDK)."""
    script = ROOT / "tools" / "lhm-server" / "build.ps1"
    if not script.exists():
        print("  SKIP: tools/lhm-server/build.ps1 not found")
        return
    result = subprocess.run(
        ["powershell", "-ExecutionPolicy", "Bypass", "-File", str(script)],
        cwd=str(script.parent),
    )
    if result.returncode != 0:
        print("WARNING: lhm-server build failed — thermals will require manual LHM install")
```

**`thermal_monitor.py`** — Minimal changes expected. If the C# server matches the reference JSON format exactly, no changes. If minor structural differences emerge during testing, `_parse_temps_from_lhm()` may need adjustment to handle both formats.

**`thermal_guidance.py`** — No changes. Receives the same data structure.

### Build pipeline

1. **Pre-build step:** `build.py` calls `build_lhm_server()` which runs `tools/lhm-server/build.ps1`. If it fails (no .NET SDK), build continues but thermals require manual LHM install — graceful degradation.
2. **PyInstaller:** Include `lhm-server.exe` as data file in `datas=[]`
3. **Runtime extraction:** PyInstaller onefile extracts all datas to `sys._MEIPASS` temp dir. The search path `os.path.join(sys._MEIPASS, "tools", "lhm-server.exe")` finds it there. Note: this is NOT alongside the exe — it's inside the `_MEIPASS` extraction directory.

**`build.ps1` skeleton:**
```powershell
# tools/lhm-server/build.ps1
$ErrorActionPreference = "Stop"
$OutDir = Join-Path $PSScriptRoot "dist"

if (-not (Get-Command dotnet -ErrorAction SilentlyContinue)) {
    Write-Error ".NET 8 SDK required. Install from https://dot.net/download"
    exit 1
}

Write-Host "Building lhm-server..."
dotnet publish -r win-x64 -c Release --self-contained `
    -p:PublishSingleFile=true `
    -p:IncludeNativeLibrariesForSelfExtract=true `
    -p:EnableCompressionInSingleFile=true `
    -o $OutDir

if (-not (Test-Path (Join-Path $OutDir "lhm-server.exe"))) {
    Write-Error "Build produced no output"
    exit 1
}

$size = (Get-Item (Join-Path $OutDir "lhm-server.exe")).Length / 1MB
Write-Host "Built: lhm-server.exe ($([math]::Round($size, 1)) MB)"
```

### Fallback chain (preserved)

```
1. Check if something is already responding on port 8085 (existing LHM instance) → attach
2. Look for bundled lhm-server.exe → launch it
3. Look for full LHM installation in standard paths → launch it
4. None found → degrade gracefully ("thermal monitoring unavailable")
```

The existing fallback chain is enhanced, not replaced. Users who already have LHM installed get the same experience. The bundled server is a new middle tier.

### Size impact

- `lhm-server.exe` (self-contained .NET 8 single-file with compression): ~10-20MB
  (LibreHardwareMonitorLib pulls in native interop for WMI, ACPI, and the kernel driver,
  making this larger than a minimal .NET console app)
- Current `lil_bro.exe`: ~20-30MB (estimated PyInstaller onefile)
- Combined: ~30-50MB — acceptable for a gaming optimization tool

### Feasibility risks

1. **Kernel driver extraction.** LibreHardwareMonitorLib uses a ring-0 kernel driver
   (`WinRing0x64.sys`) extracted to a temp directory at runtime. When `lhm-server.exe`
   is itself extracted by PyInstaller to a `_MEIPASS` temp path, the driver gets
   extracted to a nested temp dir. Potential issues:
   - Path length exceeding MAX_PATH (260 chars) on older Windows configs
   - Driver signing enforcement on Secure Boot systems
   - Antivirus flagging driver extraction from a temp directory

   **Mitigation:** Test early on a Secure Boot-enabled Win11 machine. If nested temp
   extraction fails, fall back to extracting `lhm-server.exe` to `%APPDATA%\lil_bro\tools\`
   on first run (shorter, stable path).

2. **Antivirus false positives.** PyInstaller-extracted binaries spawning .NET
   self-contained binaries that extract kernel drivers from temp dirs is a
   triple-trigger for Windows Defender and other AV engines. The existing
   `lil_bro.spec` already disables UPX for this reason.

   **Mitigation:** Test on a clean Win11 install with default Defender settings.
   Long-term: code-sign both `lil_bro.exe` and `lhm-server.exe`. Short-term:
   document Defender exclusion in the README for early adopters.

## Open Questions

1. **Port conflict handling:** If port 8085 is occupied by something other than LHM,
   should lhm-server try alternative ports (8086, 8087...) and communicate the port
   back to lil_bro? Current behavior: fail. Proposed: keep failing — port conflict is
   rare and the error message is clear.

2. **.NET 8 vs .NET Framework 4.7.2 publish target:** .NET 8 self-contained is larger
   (~10-20MB) but requires no runtime on the user's machine. .NET Framework 4.7.2 is
   smaller but requires the framework pre-installed (which most Win10/11 have).
   Recommendation: .NET 8 self-contained for zero-dependency guarantee.

3. **First-run extraction vs. PyInstaller temp:** PyInstaller extracts to a new temp dir
   on every launch. Start with `_MEIPASS` extraction (simpler). If kernel driver issues
   emerge from deep temp paths (see Feasibility Risks), switch to extracting
   `lhm-server.exe` to `%APPDATA%\lil_bro\tools\` on first run.

## Success Criteria

1. Running `lil_bro.exe` on a clean Windows install (no LHM) produces thermal readings
2. Thermal guidance in Phase 4 (esports config check) shows CPU/GPU temps
3. Pre-benchmark idle thermal gate works without manual LHM installation
4. Build size increase is under 25MB
5. No additional user steps required — thermals work on first run
6. C# server JSON output passes structural diff against `reference-output.json`
7. `lhm-server.exe` launches and responds on a Secure Boot-enabled Win11 machine
8. Windows Defender does not quarantine either binary on a clean install
