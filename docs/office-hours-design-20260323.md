## Problem Statement

Gaming PCs are routinely running at 60-70% of their hardware potential due to silent
system-level misconfigurations: monitors stuck at 60Hz when they support 144Hz+, RAM
running at base spec with XMP/EXPO disabled, Resizable BAR off, power plans set to
Balanced. The problem is **invisible** — the user doesn't know these settings exist,
so they never search for a fix. They just assume their $1,500 PC is as fast as it can be.

The status quo: if someone does discover one issue, they Google → YouTube → manually fix
one thing, missing the other five. No tool audits all system-level hardware configuration
at once.

lil_bro is a local, privacy-first hardware configuration audit tool. It's not an FPS
booster (that's Hone/Cortex territory). It's a one-time "make sure your hardware is
actually configured correctly" tool, powered by a local LLM that explains every finding
in plain English and proposes a fix plan the user approves before anything changes.

### Scope of Detection — The 6 Checks

1. **Monitor refresh rate** — Detects if active Hz < monitor's EDID-reported max Hz
2. **Mouse polling rate** — Detects if polling rate < 500 Hz (gaming threshold). Detection method: HID interval timing heuristic — measures time between cursor position updates over 3 seconds. Results may be inaccurate under system load (confidence: medium). User is always advised to verify via mouse vendor software. 500 Hz is the v1 minimum threshold; 125/250 Hz = FAIL, 1000 Hz+ = PASS.
3. **Windows Game Mode** — Detects if Game Mode is disabled (registry check)
4. **Power plan** — Detects if not set to Ultimate/High Performance
5. **XMP/EXPO (RAM speed)** — Detects if ConfiguredClockSpeed < rated Speed (WMI)
6. **Resizable BAR (ReBAR)** — Detects if GPU BAR1 memory < 256 MB indicating disabled ReBAR (heuristic — may produce false positives on unusual PCIe configurations; explanation always includes "verify in GPU-Z before acting")

Plus two additional checks:
- **Temp folder bloat:** Scans `%TEMP%`, `C:\Windows\Temp`, `SoftwareDistribution\Download`, and NVIDIA/AMD shader caches. Flags if total bloat >1 GB. Fix: delete with user approval. Always requires approval — deletion is irreversible beyond the Restore Point.
- **Cinebench 2026 baseline:** Runs single-core benchmark. Captures peak CPU/GPU temps via LHM during run. Flags thermal throttling if CPU >85°C or GPU >90°C. Binary: `Cinebench2026.exe` bundled in installer (or located in CWD). If binary is missing, benchmark is skipped with a warning.

## Demand Evidence

Community-observed pattern: r/buildapc, r/pcmasterrace, r/pcgaming repeatedly surface
users asking why their monitor isn't running at advertised Hz, why their RAM shows
2133MHz when it's rated 3600, why performance feels wrong after a fresh build. These
aren't rare edge cases — they're the default state of most Windows installs.

Key behavior: The problem is pre-discovery. Users aren't searching for a solution because
they don't know they have a problem. This makes "community-observed pain" the right
evidence category at pre-product stage. The distribution strategy must surface the
problem to people who don't know they have it.

## Status Quo

When users finally discover a single issue (usually by seeing someone else's post):
Google → YouTube → find a video for that specific setting → manually change it →
miss the other 5 issues entirely. Each issue requires its own rabbit hole. Most users
fix one thing and stop.

No existing tool runs a comprehensive hardware-level configuration audit:
- **Hone / Razer Cortex / HD OPTI:** in-game FPS optimization, cloud-connected, background process — different category entirely
- **NVIDIA App:** per-GPU settings, not system-wide config audit
- **Manual guides:** fragmented, user must self-discover issues first

lil_bro is the only local, batch, hardware-level configuration auditor.

## Target User & Narrowest Wedge

**Primary user:** The new-build gamer. Just assembled or bought a pre-built PC
($800–2000). Excited. Has XMP disabled (factory default). Monitor at 60Hz (Windows
default). ReBAR off (BIOS default). They don't know any of this.

**Broader target:** Any Windows gamer who has never run a configuration audit —
which is most of them.

**Narrowest wedge (v1):** A local terminal tool that runs a complete hardware
configuration audit, explains every finding in plain English via local LLM, and
asks for approval before fixing anything. Runs once, sets everything right.

**Distribution angle for broader reach:** The pitch isn't "new build tool" —
it's "when did you last check if your PC is actually configured right?" This reaches
any gamer, not just new builds.

## Constraints

- **Privacy:** No telemetry, no scan data, and no system information ever leaves the device. The only external call is the one-time GGUF model download on first launch (see Distribution Model). After first run, the tool is fully offline.
- Must run on Windows 10/11 (v1 scope; Mac/Linux not planned — see Hardware Coverage)
- **Requires administrator elevation** (UAC prompt on launch) — needed for LHM sensor access, Restore Point creation, power plan changes, and registry writes
- Python 3.11+ (existing codebase)
- LLM: Qwen2.5-Coder-7B-Instruct.Q4_K_M.gguf (~4.5 GB) via llama-cpp-python
- **Reference hardware for SLA testing:** Core i5-12400 / 16 GB DDR4 / NVIDIA RTX 3060 / CPU-only inference (no GPU offload assumed); LLM output capped at 512 tokens via `max_tokens` to meet timing SLA
- Every system modification requires explicit user approval (human-in-the-loop)
- System Restore Point created before any changes
- Distribution: small installer + first-run model download (not a single bundled .exe)

### Hardware Coverage (v1 Scope)

- **NVIDIA GPU:** Full support (nvidia-smi primary, WMI fallback for ReBAR)
- **AMD GPU:** Partial support — ReBAR check uses WMI fallback; GPU-specific checks degrade gracefully with "AMD detected — some checks unavailable in v1"
- **Intel integrated / no discrete GPU:** GPU-dependent checks (ReBAR, VRAM thermals) are skipped with clear messaging. All CPU/RAM/display/power checks still run.
- **Mac/Linux:** Not a v1 consideration. Architecture is intentionally Windows-native (WMI, ctypes, winreg). Cross-platform would require a separate codebase — not planned.

## Recommended Approach

**Approach B — Full product, full roadmap.**

The LLM explanation layer is what separates lil_bro from a developer script. Shipping
without it means validating the wrong product.

**Execution order (revised timeline: 6–8 weeks):**

**Phase 1 — LLM Integration (Weeks 2–3, ~2 weeks of real work)**
- **HARD GATE (Milestone 0):** Validate PyInstaller + llama-cpp-python bundle on Windows before any Phase 1 work begins. Pass criteria: a PyInstaller-bundled .exe successfully loads a GGUF model and returns an inference result on a clean Windows VM with no Python pre-installed. Do not begin Phase 2 (Hardware Integration) until this milestone passes. If it fails, switch to WinPython embeddable launcher (see Contingency).
- llama-cpp-python wrapper with CPU inference (no GPU offload for portability)
- GGUF model loading (Qwen2.5-Coder-7B-Instruct.Q4_K_M.gguf)
- JSON action proposal generation from scan results
- Human-in-the-loop approval flow (see UX spec below)

**Phase 2 — Hardware Integration (Weeks 4–5, ~2 weeks)**
- LibreHardwareMonitor: launch as background sidecar at startup, poll
  `/data.json` HTTP endpoint on localhost:8085 for CPU/GPU temp + fan RPM sensors.
  Teardown on exit. Used to: report peak temps during Cinebench run, trigger thermal
  guidance if CPU >85°C or GPU >90°C.
- **NvidiaProfileInspector:** Deferred to a future version. No v1 auto-fix action requires
  a GPU profile change, so bundling it adds trust surface with no user benefit. Will be
  added when NVIDIA-specific profile fixes are implemented.
- Thermal guidance module: compare Cinebench peak temps to thresholds, LLM generates
  plain-English guidance if throttling detected.
- **LibreHardwareMonitor failure handling:** Before launching LHM, check if port 8085 is already bound (socket probe). If occupied, warn user: "Port 8085 is in use — thermal monitoring unavailable." If LHM fails to start or `/data.json` is unreachable after 5 seconds, thermal checks are skipped with a user-visible warning. Cinebench still runs; peak temperatures are simply not reported.

**Phase 3 — Packaging (Weeks 6–7, ~1–2 weeks)**
- PyInstaller bundle (contingent on Week 2 spike success)
- Inno Setup installer
- File hash verification on launch: SHA-256 hash of the GGUF model file and all
  sidecar .exe files verified against a `manifest.json` bundled in the installer root.
  `manifest.json` is generated at build time by the CI pipeline and is version-pinned
  to each release. On mismatch: warn user, abort execution, display instructions to
  re-download. The manifest is not downloaded at runtime — it is a static build artifact.

**Contingency (if PyInstaller spike fails):** Switch to WinPython embeddable launcher — ships a self-contained Python runtime with a `.cmd` launcher. Adds ~1 week to Phase 3 due to re-testing the install flow. UX changes: double-click `.cmd` instead of `.exe`, UAC prompt looks different.

**Post-launch:** Measure downloads, gather feedback, iterate on LLM prompt quality.

## Approval UX Specification

When the LLM proposes fixes, the interaction is:

1. LLM outputs a numbered list of proposed changes in plain English
2. User sees: each finding (what was detected), its explanation (why it matters),
   and the proposed action (what will change)
3. User types the number(s) of changes to apply, or "all", or "skip" — one prompt,
   batch selection
4. Each selected change executes with a success/fail confirmation line
5. On partial failure: user is informed, advised to use the Restore Point if needed.
   No automatic rollback beyond the system-level Restore Point created at startup.

Example terminal output:
```
[1] Monitor refresh rate: 60Hz detected, 144Hz available
    Fix: Change Windows display refresh rate to 144Hz
[2] RAM speed: 2133MHz running, 3600MHz rated (XMP off)
    Fix: (BIOS change required — lil_bro will show you how)
[3] Power plan: Balanced → Ultimate Performance
    Fix: Switch power plan via powercfg

Apply changes? Enter numbers (e.g. "1 3"), "all", or "skip":
```

## LLM JSON Contract

The LLM receives scan results as structured JSON and must return a JSON array of
action proposals. This is the contract engineers implement against.

**System prompt (draft):**
```
You are a gaming PC configuration assistant. You will receive a JSON object with
hardware info and a list of configuration findings. For each FAIL finding, output a
JSON array of action proposals. Use casual, direct language — write like a knowledgeable
friend, not a manual. Do not mention checks with PASS status. Respond ONLY with a valid
JSON array — no explanation text outside the JSON. Keep each explanation under 40 words.
```

**Input (system scan summary):**
```json
{
  "hardware": { "cpu": "Intel Core i5-12400", "gpu": "NVIDIA RTX 3060", "ram_gb": 32 },
  "findings": [
    { "check": "display_hz", "current": 60, "max_available": 144, "status": "FAIL" },
    { "check": "xmp", "current_mhz": 2133, "rated_mhz": 3600, "status": "FAIL" },
    { "check": "power_plan", "current": "Balanced", "status": "FAIL" },
    { "check": "game_mode", "enabled": false, "status": "FAIL" },
    { "check": "mouse_polling_hz", "current": 125, "threshold": 500, "status": "FAIL" }
  ]
}
```

**Expected output (action proposals):**
```json
[
  {
    "finding": "display_hz",
    "severity": "HIGH",
    "explanation": "Your monitor supports 144Hz but Windows is running at 60Hz. You're seeing less than half the frames your hardware can produce.",
    "proposed_action": "Change display refresh rate to 144Hz via Windows Display Settings",
    "can_auto_fix": true
  },
  {
    "finding": "xmp",
    "severity": "HIGH",
    "explanation": "Your RAM is rated for 3600MHz but running at 2133MHz. XMP is off by default in your BIOS — costs 10-15% CPU performance.",
    "proposed_action": "Enable XMP/EXPO in BIOS (manual — lil_bro will show step-by-step instructions)",
    "can_auto_fix": false
  },
  {
    "finding": "game_mode",
    "severity": "MEDIUM",
    "explanation": "Windows Game Mode is off. It prioritizes CPU/GPU resources to your game and reduces background jitter.",
    "proposed_action": "Enable Game Mode via registry write to HKCU\\SOFTWARE\\Microsoft\\GameBar",
    "can_auto_fix": true
  },
  {
    "finding": "mouse_polling_hz",
    "severity": "MEDIUM",
    "explanation": "Your mouse is reporting at 125Hz — way too slow for competitive gaming. Most gaming mice support 1000Hz+ in their driver software.",
    "proposed_action": "Open your mouse's driver software and set polling rate to 1000Hz or higher (manual — varies by manufacturer)",
    "can_auto_fix": false
  }
]
```

**Field semantics:**
- `severity`: HIGH / MEDIUM / LOW — displayed as a label in the approval UI; findings are sorted HIGH first. No effect on approval flow logic.
- `can_auto_fix`: true = lil_bro can apply this programmatically; false = requires manual user action (shown as informational only, no approval prompt)
- `approval_required` field is omitted from the LLM output — it is **always** implied true for `can_auto_fix: true` items and handled by the approval UX layer. The LLM does not decide approval requirements.

**LLM output failure handling:** If the model returns invalid or non-parseable JSON, the tool logs the raw output, displays "AI explanation unavailable — showing raw findings instead," and falls back to a static summary of each FAIL check. It does not crash. Retry once before falling back.

**Error states (per-check failure behavior):**
| Check | If tool/API unavailable | Behavior |
|---|---|---|
| display_hz | WMI unavailable | Skip + warn: "Display check unavailable — WMI unreachable" |
| xmp | WMI unavailable | Skip + warn |
| rebar | nvidia-smi missing | Fallback to WMI; if WMI also fails, skip + warn |
| game_mode | Registry read fails | Skip + warn |
| power_plan | powercfg missing | Skip + warn |
| mouse_polling | Win32 API fails | Skip + warn |
| thermals | LHM unavailable | Skip + warn; Cinebench still runs |
| cinebench | Binary not found | Skip benchmark entirely; warn user |

If 3 or more checks fail to run, prompt user: "Limited scan results — some checks couldn't run. Results may be incomplete." LLM still receives partial results and generates proposals for available findings only.

## Distribution Model (addressing 4.5 GB / GitHub Releases limit)

GitHub Releases has a 2 GB per-asset limit. The GGUF model alone is ~4.5 GB.
Options evaluated:

**Chosen approach: Staged download on first run**
- The installer (~50 MB) installs the detection engine and LHM sidecar
- On first launch, lil_bro informs the user that the GGUF model (~4.5 GB) must be downloaded, then downloads it from HuggingFace Hub or Backblaze B2
- Model is cached in `%APPDATA%\lil_bro\models\`; subsequent runs are instant
- Pros: installer is small, fast to download, easy to share; model download is a one-time cost
- Cons: requires internet for first run (breaks offline-first for first launch only)

**Privacy note on model download:** The download is a standard HTTPS GET request — no scan data or system identifiers are sent. The user's IP is exposed to the model host (HuggingFace/Backblaze), which is disclosed to the user before the download begins. No scan results or hardware info is transmitted.

## Open Questions

1. **Model size vs. quality (BLOCKING):** Benchmark Qwen2.5-3B-Instruct Q4_K_M (~1.8 GB) vs 7B on reference hardware before committing to 7B. If the 7B model exceeds the 60-second SLA on reference hardware (i5-12400, CPU-only, max_tokens=512), fall back to 3B — do not ship with a broken SLA. This must be resolved before Phase 1 LLM integration begins.

2. **Trust / AV false positives:** PyInstaller .exe files frequently trigger Windows
   Defender. Mitigations: code-signing certificate (recommended), GitHub releases page
   with VirusTotal scan link, clear open-source repo.

3. **Distribution channel (launch plan):** Without an existing audience, how do the first
   1,000 downloads happen? Concrete options: r/buildapc post, r/pcgaming post, YouTube
   demo video (3-min screen recording showing a fresh build audit), tech influencer outreach.

4. **Monetization (Future Considerations — not v1 scope):** Free tool for validation.
   Post-validation options include paid license, Pro tier with scheduled audits or driver
   integration. This is explicitly deferred — do not let it drive v1 architecture decisions.

## Success Criteria

- Installer .exe: <100 MB; total installed footprint after model download: <5 GB

- LLM generates explanation + fix proposal in <60 seconds on reference hardware
  (i5-12400 / 16 GB DDR4 / CPU-only inference, max_tokens=512 enforced)

  shown as informational only

**Post-launch (30 days):**
- 500+ downloads
- At least 10 user reports of "settings were wrong, now fixed"
- At least 1 community post organically sharing the tool
